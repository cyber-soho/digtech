🔵 LESSON 1

Augmented Reality (AR)

"Augmented Reality doesn't replace the world. It enriches it."

1.1 What is AR?

Augment means to make something greater by adding to it. So AR is the practice of layering computer-generated content — images, sounds, text, 3D models, data overlays — on top of the real world as you experience it through a lens: your phone camera, a tablet, or a pair of smart glasses.

You are not transported somewhere else. You are here, but here has been upgraded. The key distinguishing factor of AR is that the physical environment remains the primary experience. The digital layer is additive — it exists to provide context, information, or enhancement, not to replace what is already visible.

Think of it like this: imagine you're a doctor looking at a patient's arm. With AR, a digital overlay could show you the exact location of the vein beneath the skin before you insert a needle. The arm is still real. The overlay is digital. Together, they create a richer experience that results in better outcomes.

1.2 The Technical Architecture of AR

There are four technical pillars that every AR system — from a simple phone app to a military heads-up display — relies on:

📷

SENSING

Camera, IMU, GPS

→
🗺️

TRACKING

SLAM, GPS, Markers

→
🎨

RENDERING

3D Models, Text, Holograms

→
🥽

DISPLAY

Screen, Glass, Projector

Pillar 1 — Sensing: Hardware captures the real world. Cameras capture visual data. IMU sensors track orientation. GPS provides geographic position.
Pillar 2 — Tracking: SLAM (Simultaneous Localisation And Mapping) builds a map of the environment and locates the device within it — simultaneously — in real time.
Pillar 3 — Rendering: Digital content is generated — 3D models, text labels, animated characters, data charts — and mathematically aligned to real-world coordinates.
Pillar 4 — Display: The composited image (real world + digital layer) is shown to the user, either on a screen or through transparent optics in a headset.

💻 Code Example — lesson1_ar_inventory_overlay.py

lesson1_ar_inventory_overlay.py 
# ============================================================
# LESSON 1 — AR INVENTORY OVERLAY SIMULATION
# File: lesson1_ar_inventory_overlay.py
# Requires: pip install pillow numpy
# ============================================================

from PIL import Image, ImageDraw, ImageFont
import numpy as np
import random
from datetime import datetime

# STEP 1: Define our "inventory database"
# In a real system this would be a SQL or NoSQL database call.

INVENTORY_DB = {
  "item_A": {"name": "Organic Apples",   "stock": 40, "expiry": "2025-09-10", "alert": False},
  "item_B": {"name": "Full-Fat Milk",    "stock":  5, "expiry": "2025-08-01", "alert":  True},
  "item_C": {"name": "Sourdough Bread", "stock": 22, "expiry": "2025-07-29", "alert": False},
  "item_D": {"name": "Cheddar Cheese",  "stock":  3, "expiry": "2025-08-15", "alert":  True},
}

# STEP 2: Simulated object detections
# Format: (item_id, x1, y1, x2, y2) — bounding box corners

DETECTED_OBJECTS = [
  ("item_A",  30,  80, 200, 220),
  ("item_B", 240,  80, 410, 220),
  ("item_C", 450,  80, 620, 220),
  ("item_D", 660,  80, 830, 220),
]

# STEP 3: Create simulated camera frame
CANVAS_WIDTH, CANVAS_HEIGHT = 900, 350
base_image = Image.new("RGB", (CANVAS_WIDTH, CANVAS_HEIGHT), color=(210, 210, 210))
draw = ImageDraw.Draw(base_image)

# STEP 4: Draw physical shelf
draw.rectangle([20, 230, 880, 250], fill=(139, 90, 43))   # brown shelf
draw.rectangle([20,  50, 880, 230], fill=(245, 245, 220))  # off-white wall

# STEP 5: Draw physical boxes
for item_id, x1, y1, x2, y2 in DETECTED_OBJECTS:
    draw.rectangle([x1, y1, x2, y2], fill=(160, 120, 60), outline=(80, 50, 20), width=2)
    draw.text((x1 + 5, y2 - 20), f"SKU:{item_id[-1]}", fill=(255, 255, 255))

# STEP 6: THE AR OVERLAY RENDERING LOOP — The Magic
for item_id, x1, y1, x2, y2 in DETECTED_OBJECTS:
    item_data = INVENTORY_DB.get(item_id)
    if not item_data: continue

    overlay_colour = (220, 50, 50) if item_data["alert"] else (30, 140, 60)
    centre_x = (x1 + x2) // 2
    panel_x1, panel_y1 = x1 - 5, y1 - 75
    panel_x2, panel_y2 = x2 + 5, y1 - 5

    draw.rectangle([panel_x1, panel_y1, panel_x2, panel_y2],
                   fill=(20, 20, 20), outline=overlay_colour, width=2)
    draw.text((panel_x1+5, panel_y1+5),  item_data["name"], fill=(255,255,255))

    stock_text = f"Stock: {item_data['stock']} units"
    if item_data["alert"]: stock_text += " ⚠ LOW"
    draw.text((panel_x1+5, panel_y1+22), stock_text, fill=overlay_colour)
    draw.text((panel_x1+5, panel_y1+39), f"Exp: {item_data['expiry']}", fill=(200,200,200))
    draw.line([(centre_x, panel_y2), (centre_x, y1)], fill=overlay_colour, width=1)

# STEP 7: HUD overlay — timestamp and AR mode
timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
draw.rectangle([0, 310, CANVAS_WIDTH, 350], fill=(15, 15, 15))
draw.text((10, 318), f"AR MODE ACTIVE | {timestamp} | ITEMS TRACKED: {len(DETECTED_OBJECTS)}",
          fill=(0, 220, 0))

# STEP 8: Save output
base_image.save("lesson1_ar_output.png")
print(f"✅ AR overlay simulation complete! Items tracked: {len(DETECTED_OBJECTS)}")

📖 Step-by-Step Explanation

INVENTORY_DB — The Data Backend

Think of this as a simplified version of a real warehouse inventory system like those used by Amazon or IKEA. Each product has a unique ID, a human-readable name, a stock count, and an alert flag for critical stock. In a real AR deployment, this database is queried via an API call over Wi-Fi every time the camera detects a new object.

DETECTED_OBJECTS — Simulated Bounding Boxes

Real AR systems use computer vision models like YOLOv8 or DETR to detect objects in each camera frame and return bounding boxes. We hard-code these here to focus on the overlay logic. The bounding box coordinates (x1, y1, x2, y2) tell us where on the image the physical object appears.

Creating the Camera Frame

We create a blank image using Pillow and draw a warehouse scene onto it. In production, this step is replaced by capturing a live frame from a physical camera attached to the AR headset — arriving 30–60 times per second.

Drawing the Physical World

We draw the shelf and boxes that represent what the camera actually sees — the real world component. Colours are chosen to be realistic: brown cardboard boxes, off-white wall. This is the layer onto which digital content will be composited.

⭐ The AR Overlay Loop — The Magic

This is the core of the AR system. For each detected object, we look up its data, choose an appropriate colour (red for alerts, green for normal), calculate where to position the floating label above the physical box, and draw text panels with name, stock count, and expiry.

The vertical anchor line is a UI pattern used in real AR headsets to visually connect floating labels to physical objects. This exact pattern is used in Microsoft HoloLens industrial applications today.

HUD Overlay & Output

A HUD (Heads-Up Display) is always present in professional AR systems. It shows the operator what mode the system is in, the current time, and how many items are being tracked. Military pilots have used HUD displays since the 1950s; AR brings this concept to industrial workers. In a live AR system, Step 8 would push the frame to the display at 60 FPS instead of saving it.

🟣 LESSON 2

Virtual Reality (VR)

"VR is the ultimate empathy machine. It doesn't just show you a story — it puts you inside one." — Chris Milk

2.1 What is VR?

Where AR enriches the real world, VR (Virtual Reality) replaces it entirely. Put on a VR headset and your brain receives a completely different set of visual, auditory, and sometimes haptic signals. Your eyes see only the virtual world rendered inside the headset's displays. The physical room you are standing in simply ceases to exist from your perceptual standpoint.

The experience is called immersion — and achieving true immersion is both an art and a deeply technical science.

VR is not new. The concept was first formally described by Ivan Sutherland at MIT in 1965, in a paper he called "The Ultimate Display". He described a room in which the computer could control the existence of matter. Sixty years later, we're still working on it — but we are remarkably close.

2.2 The Three Pillars of Immersion

PILLAR 01
PRESENCE

The feeling of "being there"

  • Wide FOV (100°+) to match human vision
  • Low latency (< 20ms Motion-to-Photon)
  • High frame rate (72–120 FPS required)
  • Head tracking (6DOF — Six Degrees of Freedom)
PILLAR 02
INTERACTIVITY

The ability to act on the virtual world

  • Hand controllers (6DOF tracked)
  • Hand tracking (camera-based)
  • Eye tracking (gaze-based selection)
  • Physics simulation (Bullet, PhysX)
PILLAR 03
PLAUSIBILITY

The sense that the rules make sense

  • Spatial audio (3D positional sound)
  • Consistent physics (gravity, collisions)
  • Believable avatars and NPC AI
  • Haptic feedback (vibration, resistance)
Studies from Stanford University's Virtual Human Interaction Lab have shown that VR experiences produce measurable changes in real-world behaviour — people who experience poverty in VR are measurably more empathetic. People who see themselves ageing in VR save more money for retirement. This is not entertainment. This is applied neuroscience.

💻 Code Example — lesson2_vr_headtracking.py

lesson2_vr_headtracking.py 
# ============================================================
# LESSON 2 — VR HEAD-TRACKING SIMULATION
# File: lesson2_vr_headtracking.py
# Requires: pip install numpy matplotlib
# ============================================================

import numpy as np
import matplotlib.pyplot as plt
from mpl_toolkits.mplot3d import Axes3D
import math

# STEP 1: Euler angles → 3×3 Rotation Matrix
# This conversion runs in every VR headset's tracking firmware.

def euler_to_rotation_matrix(pitch_deg, yaw_deg, roll_deg):
    """Convert pitch, yaw, roll (degrees) to 3×3 rotation matrix.
    Same conversion run in every VR headset's tracking firmware."""
    p = math.radians(pitch_deg)
    y = math.radians(yaw_deg)
    r = math.radians(roll_deg)

    Rx = np.array([[1,0,0], [0,math.cos(p),-math.sin(p)], [0,math.sin(p),math.cos(p)]])
    Ry = np.array([[math.cos(y),0,math.sin(y)], [0,1,0], [-math.sin(y),0,math.cos(y)]])
    Rz = np.array([[math.cos(r),-math.sin(r),0], [math.sin(r),math.cos(r),0], [0,0,1]])

    return Rz @ Ry @ Rx  # ZYX Euler convention — standard in VR systems

# STEP 2: Simulate 120 timesteps of IMU readings (2s at 60Hz)
NUM_TIMESTEPS = 120
imu_readings = []

for t in range(NUM_TIMESTEPS):
    progress = t / NUM_TIMESTEPS
    pitch = 15.0 * math.sin(2 * math.pi * progress)   # ±15° tilt
    yaw   = 60.0 * math.sin(math.pi * progress)         # 0→60° panoramic turn
    roll  =  5.0 * math.sin(4 * math.pi * progress)   # subtle natural roll
    imu_readings.append((t, pitch, yaw, roll))

# STEP 3: Compute look-direction vectors from rotation matrices
look_vectors = []
for t, pitch, yaw, roll in imu_readings:
    R = euler_to_rotation_matrix(pitch, yaw, roll)
    default_forward = np.array([0, 0, -1])  # OpenGL convention: -Z is forward
    look_vectors.append(R @ default_forward)

# STEP 4: Analytics — data scientist domain
lv_array = np.array(look_vectors)
yaw_values   = [r[2] for r in imu_readings]
pitch_values = [r[1] for r in imu_readings]
max_yaw   = max(abs(v) for v in yaw_values)
max_pitch = max(abs(v) for v in pitch_values)

print(f"Timesteps: {NUM_TIMESTEPS} | Duration: {NUM_TIMESTEPS/60:.1f}s at 60Hz")
print(f"Peak yaw: ±{max_yaw:.1f}° | Peak pitch: ±{max_pitch:.1f}°")

# STEP 5: 3D Visualisation of head tracking path
xs, ys, zs = lv_array[:,0], lv_array[:,1], lv_array[:,2]

fig = plt.figure(figsize=(12, 8))
ax = fig.add_subplot(111, projection='3d')

for i in range(len(xs) - 1):
    colour = (i / len(xs), 0, 1 - i / len(xs))  # Blue→Red over time
    ax.plot(xs[i:i+2], ys[i:i+2], zs[i:i+2], color=colour, linewidth=2)

ax.scatter([xs[0]],  [ys[0]],  [zs[0]],  color='green', s=100, label='Start')
ax.scatter([xs[-1]], [ys[-1]], [zs[-1]], color='red',   s=100, label='End')
ax.set_title("VR Head Tracking Path — Look Direction Vector Over Time")
plt.savefig("lesson2_vr_tracking_plot.png", dpi=150)

📖 Step-by-Step Explanation

euler_to_rotation_matrix() — The Core Math

Euler angles (pitch, yaw, roll) are how humans intuitively describe head orientation. But computers and GPUs need this expressed as a 3×3 rotation matrix for efficient matrix multiplication during rendering. This function performs that exact conversion. Every VR headset does this conversion millions of times per second in its tracking firmware, written in C++ for speed.

Simulating IMU Sensor Data

Real VR headsets have physical IMU chips (e.g., InvenSense ICM-42688-P) outputting data at 1000Hz+. We simulate 120 readings (2 seconds at 60Hz) using sinusoidal functions representing natural head movement. The sinusoidal shape is important — real head movements are smooth curves, not sharp jumps.

Computing Look Direction Vectors

For each IMU reading, we apply the rotation matrix to a "default forward vector" [0, 0, -1] — the OpenGL convention for looking straight ahead. After applying rotation, we have the actual direction the user is looking at that moment in the virtual world.

Analytics — The Data Scientist's Domain

VR platforms collect vast amounts of head and gaze tracking data. By analysing peak rotation angles, time spent looking in each direction, and movement patterns, platform developers can understand user engagement. Companies like Meta and Unity provide analytics dashboards built on exactly this type of raw tracking data.

3D Visualisation — Gaze Heatmaps

The 3D plot shows the path of the look direction vector on the surface of a unit sphere. The colour gradient from blue to red encodes time progression. This type of visualisation is used by VR UX designers to understand where users look in a scene — equivalent to an eye-tracking heatmap, but in three dimensions.

🌐 LESSON 3

A New UI — XR (Extended Reality)

"XR is not a technology. It is a new way of thinking about the interface between human beings and information."

3.1 What is XR?

XR (Extended Reality) is the umbrella term that encompasses AR, VR, and MR (Mixed Reality). It refers to all technologies that blend or replace the real world with digital content. The "X" in XR is intentionally generic — it is a variable placeholder that represents any level of reality modification, from a subtle data overlay all the way to complete digital immersion.

But XR is more than just a vocabulary term. It represents a paradigm shift in UI design — a complete rethinking of how human beings interact with information and computers.

3.2 Milgram's Reality–Virtuality Continuum (1994)

REAL REALITY

3.3 MR (Mixed Reality) — The Middle Ground

In AR, digital objects are overlaid onto the real world but do not interact with it. In MR, digital objects are spatially anchored to the real world and interact with physical surfaces in real time.

Example: In AR, a 3D model of a chair just floats on your screen. In MR, the chair sits on your actual floor, casts a virtual shadow on it, and you can walk around it — it stays in place as you move.

Apple Vision Pro (launched 2024) and Microsoft HoloLens 2 are the most prominent MR devices. They use depth cameras and SLAM to precisely map the physical environment and anchor digital content to it.

3.4 XR as a New Paradigm in UI Design

Up until now, every interaction with a computer has been fundamentally flat. Screen → keyboard → mouse. Even touchscreens are flat. The entire history of GUI design — from Xerox PARC in the 1970s, through Windows, macOS, iOS, Android — has been constrained to a rectangle.

XR shatters that constraint. In an XR environment, the UI is spatial. Information exists in three-dimensional space around you. You reach out and grab a file. You turn your head to switch applications. You place your data dashboard on the wall to your left.

This is what Apple calls Spatial Computing — and it represents the most significant shift in human-computer interaction since the introduction of the touchscreen.

3.5 The Five Principles of Spatial UI Design

01

DEPTH IS INFORMATION

Objects closer to the user feel more urgent/important. Use Z-axis distance to convey hierarchy and priority.

02

THE BODY IS THE CURSOR

Eyes, head pose, hands, and voice replace mouse and keyboard. Design for gaze-based hover, pinch-to-select, and voice commands.

03

COMFORT ZONES MATTER

Keep interactive elements within the user's comfortable reach zone (0.5m–2m). Content below 0.5m or above 60° elevation causes neck strain.

04

CONTEXT COLLAPSES MENUS

In XR, menus are contextual — they appear near the relevant object. The concept of a "menu bar" disappears; interaction is in-place.

05

THE ENVIRONMENT IS THE CANVAS

Walls, floors, tables, and surfaces become valid UI real estate. A data visualisation can be as large as the room demands.

students.forEach(student => {

// MOUSE DRAG → 3D ROTATION (Spatial UI Principle 2: The Body is the Cursor)
document.addEventListener('mousemove', e => {
  if (!isDragging) return;
  rotY += (e.clientX - prevX) * 0.5;  // left/right drag = yaw
  rotX += (e.clientY - prevY) * 0.5;  // up/down drag = pitch

// AUTO-ROTATION — ambient animation shows 3D depth when idle
setInterval(() => {

}, 50);

📖 Step-by-Step Explanation

The Data Layer

The students array is our synthetic dataset — 60 records with score, study hours, and assignment completion rate. In a real XR application, this data would come from a university LMS (Moodle or Canvas) via a REST API call. Each student's category is derived from their score using thresholds that match the CATEGORIES object.

The 3D Positioning Logic

The core innovation: mapping data dimensions to spatial coordinates — study hours → X, score → Y, completion rate → Z. This is what makes the scatter plot spatial rather than flat. The Z-axis (depth) provides a third data dimension that flat charts cannot show. This directly implements Spatial UI Principle 1: Depth is Information.

The Tooltip System — Gaze Simulation

Hovering over a data point triggers a contextual overlay showing that student's full record. This simulates gaze-triggered contextual menus in XR — Spatial UI Principle 4. In a real XR headset, the trigger would be eye gaze dwelling on a data point for 300ms rather than a mouse hover.

The Drag Rotation System

Mouse drag maps to 3D rotation of the entire data cloud. Left-right drag = Y axis (yaw); up-down drag = X axis (pitch). These are exactly the same axes from Lesson 2's head tracking discussion. The user is the cursor — Spatial UI Principle 2: The Body is the Cursor.

The Filter System

Category filtering demonstrates how XR interfaces handle data exploration. Rather than navigating to a different screen or opening a menu, the user gestures to filter — matching data points glow while others fade. The result is always presented spatially, never as a list or new page — demonstrating Principle 4: Context Collapses Menus.

Meta Horizon OS

Open ecosystem for spatial computing — Quest 3 as the affordable entry point for consumers and developers.

WebXR (W3C)

Standard for XR in web browsers — any website could become an XR application without app installation.

OpenXR (Khronos)

Open, royalty-free standard for VR/AR app and device interoperability — adopted by most major XR platforms.

NVIDIA Omniverse

Platform for 3D simulations and collaborative virtual environments — backbone of industrial metaverse applications.
